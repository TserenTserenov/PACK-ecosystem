# [ECO.SYS.001] Лид

> **Статус: draft.** Записан на Ф1 WP-251 (консенсус пир-сессии `2026-07-10-32-wp251-ontology-types`). Источник состояний — доклад Алёны (WP-233) + граничный кейс из token-level воплощения WP-385.

## Owner

Алёна Чайковская (`DP.ROLE.082`) — доменное содержание (типы Лидов, причины незавершённого перехода). Структуру системы (эту схему, новые состояния верхнего уровня) меняет только через R5 Архитектор.

## Instance level

`instance_level: yes` — Лид имеет экземпляры в БД Neon (мост Pack↔БД — Ф2 WP-251, dep WP-228).

## States

| # | Состояние | Описание | Точка входа/выхода |
|---|-----------|----------|---------------------|
| 1 | **холодный** | Не контактировал с экосистемой | Вход системы (создание Лида) |
| 2 | **прошёл первый контакт/диагностику** | Контактировал (например, прошёл бота-навигатора / Диагноста R28), не купил | Переход `1→2` — точка приложения метода верификации коммуникации |
| 3 | **покупатель** *(граничное)* | Принял решение о покупке/вступлении | Переход `2→3` = выход из ECO.SYS.001 (метод `ECO.M.022-lead-to-buyer`) |

Состояние 3 — **граничное** (см. Boundary states ниже): экземпляр покидает систему «Лид» и попадает в промежуточную зону перед `ECO.SYS.002`/`ECO.SYS.003`.

## Boundary

**В систему входит:** любой контакт с экосистемой до факта покупки/вступления — независимо от канала (маркетинг WP-188, наставник 1:1, органика).

**В систему НЕ входит:**
- Покупатель после решения о покупке — это уже вне ECO.SYS.001 (см. Boundary states).
- Член сообщества (`ECO.SYS.003`) — другая система, другая роль, другой набор методов.
- Причины отказа/оттока Лида как отдельная типология — Ф5 WP-251 (backlog, метрики конверсии).

## Boundary states

**«Покупатель» (выходное состояние метода `ECO.M.022-lead-to-buyer`).** Именованная граница, НЕ отдельная Pack-система — вопрос закрыт на Ф2 WP-251 (11.07.2026). Гап-матрица моста Pack↔БД (`07-map/ECO.MAP.BIND.001-pack-to-db.md §Гап-матрица, п.1`) не нашла в Neon таблицы «покупатель» без естественного места в `ECO.SYS.001/002/003`: переход реализуется composite-событием `#11 lead.CLAIM` + `#3 payment.PAYMENT(succeeded)`, немедленно сменяемым `#4 subscription.CONTRACT`. `ECO.SYS.004-buyer` не создаётся.

Симметрия «3 системы-объекта (Лид, Сообщество, Участник) + 2 конвейера», заложенная на Ф0 (`ECO.SC.003`), подтверждена эмпирически на Ф2, а не только сохранена умозрительно на Ф1.

## Invariants

1. Каждый Лид, вошедший в конвейер продвижения, имеет прослеживаемое состояние на всём цикле конвейера (см. `ECO.SC.003` §Инвариант).
2. Переход `холодный → прошёл первый контакт` не может произойти без явного метода взаимодействия (маркетинговый триггер, наставник, органика) — Лид не «сам себя» переводит.
3. Переход `2→3` (в покупателя) необратим внутри ECO.SYS.001 — назад в «холодный»/«тёплый» Лид не возвращается; повторный интерес после отказа = новый экземпляр Лида или явная реактивация (не формализовано на Ф1, кандидат Ф5).

## Scenarios

1. **Лид проходит бота-навигатора, не покупает Марафон.** Состояние: холодный → прошёл первый контакт/диагностику. Остаётся в состоянии 2 неопределённо долго (граничный кейс token-level WP-385, уровень 0).
2. **Наставник Марафона ведёт 1:1-переписку с Лидом.** Перед отправкой сообщения — опциональный прогон через `ECO.M.024-lead-persona-verification` (см. `ECO.SC.003` сценарий 3).
3. **Лид принимает решение о покупке Марафона.** Переход `2→3`, выход из ECO.SYS.001 через `ECO.M.022-lead-to-buyer`, попадает в граничное состояние «покупатель».
