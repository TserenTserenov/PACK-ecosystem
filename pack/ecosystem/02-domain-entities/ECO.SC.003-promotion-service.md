# [ECO.SC.003] Лид доводится до решения о покупке/вступлении с прослеживаемым состоянием

> **Статус: draft.** Записан по итогам Ф0 IntegrationGate WP-251 (консенсус пир-сессии `2026-07-10-10-ds-my-strategy-inbox`). Точные состояния/переходы Лида, словарь причин отказа и метрики конверсии — предмет Ф1 WP-251 (`ECO.SYS.001-lead.md`), здесь зафиксирован только уровень обещания.

## Обещание

**Кому:** Лиду — быть доведённым до ясного решения, а не потерянным в процессе; Алёне (Руководитель продвижения) — управляемый, наблюдаемый конвейер вместо чёрного ящика.

**Зачем:** доклад Алёны (19 апр 2026, источник WP-233) артикулировал 5 систем службы продвижения, но они не формализованы в Pack — из-за этого нельзя автоматизировать конвейеры (WP-155), связать с подпиской (WP-194), передать Алёне владение доменом без утечки архитектурных решений в операционку.

**Что получит:**
- Лид входит в конвейер продвижения с зафиксированной точкой входа.
- На выходе — один из двух исходов: переход в Покупателя (метод перехода — предмет Ф1) ИЛИ Лид остаётся Лидом с явной причиной незавершённого перехода.
- Ни один Лид не «теряется» молча — застревание видно (конкретный механизм наблюдаемости — Ф1/Ф2, мост к БД).

**Инвариант:** каждый Лид, вошедший в конвейер, имеет прослеживаемое состояние и хотя бы одну попытку перевода за цикл конвейера.

**Критерий приёмки (черновой, точные цифры — Ф1):** конверсия Лид→Покупатель отслеживается; доля Лидов без единой попытки перевода за цикл конвейера = 0.

## Контроль качества коммуникации

Материалы конвейера (посты, письма, 1:1-сообщения наставников), адресованные Лиду, **рекомендуется** (`may`, не обязательный гейт) прогонять через метод `ECO.M.NNN-lead-persona-verification` до публикации — адверсариальная проверка через банк персон-скептиков Лида (`DS-my-strategy/inbox/WP-251/personas.md`, постоянный дом решается на Ф1 WP-251, подшивка к `ECO.SYS.001`).

Метод не входит в инвариант или режим отказа этого SC: отказ прогона не блокирует публикацию, находки метода — список рекомендаций автору, не контракт. Материализация метода как отдельного `ECO.M.NNN` — Ф1 WP-251 (нет смысла заводить файл метода до появления `ECO.SYS.001`, к которому он подшивается).

Параметр «тип коммуникации» (пост / письмо / 1:1-сообщение) — один метод с разными шаблонами прогона, не отдельные методы на канал.

## Сценарии использования

1. **Алёна перед публикацией поста/письма конвейера.** Читает подготовленный промо-материал → опционально прогоняет через банк персон-скептиков (`ECO.M.NNN-lead-persona-verification`, параметр «тип коммуникации = пост») → публикует или правит по находкам.
2. **Агент-автор контента (R4 Автор) готовит черновик по брифу конвейера.** Перед сдачей Алёне на приёмку самостоятельно прогоняет черновик через тот же метод (параметр «тип коммуникации = пост/письмо») — снижает долю правок на стороне Алёны.
3. **Наставник Марафона готовит 1:1-сообщение конкретному Лиду.** Тот же метод, параметр «тип коммуникации = личное сообщение» — не отдельный метод под канал, только другой шаблон прогона (решение Ф0: дробление по каналам не оправдано).

## Не покрывает

- Материализацию метода `ECO.M.NNN-lead-persona-verification` как отдельного Pack-артефакта — Ф1 WP-251.

- Конвейер продюсирования (участник → известный) — отдельная система из 5 артикулированных Алёной, не входит в этот SC.
