# [ECO.SC.002] Возврат за X (полный или частичный)

## Обещание

**Кому:** Плательщику (пилот, корпоративный заказчик), Sales-менеджеру (оформляет ручной refund), Бухгалтеру (фиксирует возврат), Финансовому контролёру.

**Зачем:** Прозрачное правило возврата средств за установленное окно времени X. Защищает плательщика от запертого платежа за неполученную услугу и одновременно ограничивает злоупотребления.

**Что получит:**
- Возможность запросить возврат **полной суммы** в течение окна X₁ (полный refund, типично X₁ = 14 дней с момента оплаты, для подписки = grace-период 7 дней).
- Возможность запросить **частичный возврат** в окне X₂ > X₁ при выполнении дополнительных условий (например, неиспользованная часть подписки за оставшиеся месяцы).
- Прозрачный канал запроса (`/cancel` в боте, форма в кабинете, обращение к Sales-менеджеру для B2B).
- Уведомление о статусе обработки запроса в течение N банковских дней.

**Инвариант:** Возврат возможен по правилам:
- **Подписка (ECO.D.010):** полный refund в течение grace-периода (например, 7 дней с момента списания); далее — pro-rata за неиспользованные циклы либо отказ (по политике).
- **Покупка (ECO.D.010):** полный refund в окне X₁ при отсутствии «потребления» товара; иначе — отказ или частичный.
- **Донат (ECO.D.010):** возврат на усмотрение получателя, без обязательства.

Окна X₁/X₂ фиксируются конкретными значениями в политике возврата и оферте; здесь определяется только структура SC, не конкретные значения.

**Критерий приёмки:** Запрос на refund обрабатывается в течение N банковских дней. Решение (approve / partial / decline) фиксируется с причиной. Возвращённые средства зачисляются обратно тем же каналом, что и платёж.

## Основной flow

```
Плательщик → Запрос refund (бот /cancel, форма, обращение к Sales)
       │
       ▼
Проверка окна X₁ / X₂  ─────► Окно истекло ─► Отказ с обоснованием
       │
       ▼
Проверка причины (опц.) ─────► Неприемлемая ─► Отказ
       │
       ▼
Расчёт суммы:
   - полный refund (в X₁) → 100% от платежа
   - частичный (в X₂)    → pro-rata по правилу
       │
       ▼
Обработка через платёжный канал ─► Списание у получателя ─► Зачисление плательщику
       │                                                       (T+N банковских дней)
       ▼
Обновление статуса:
   - ECO.D.011 cancel-событие (для подписки)
   - finance.payments: статус "refunded" / "partial-refund"
   - ECO.D.012: бонусы по правилам балансов, не denomination в фиат
       │
       ▼
Уведомление плательщику: "возврат N руб. зачислен на канал Y, ETA T+M дней"
```

## Failure modes

| Класс | Симптом | Что происходит | Recovery |
|-------|---------|---------------|----------|
| **Просроченное окно** | Запрос пришёл позже X₁ (или X₂ для частичного) | Автоматический отказ с указанием даты истечения окна | Эскалация к Sales-менеджеру для исключений по B2B-договору |
| **Неверный канал** | Платёж был через канал A, возврат запрашивается на канал B | Отказ — возврат только по тому же каналу (требование платёжных систем) | Уведомить плательщика; повторный запрос на исходный канал |
| **Частичный возврат с бонусами** | Часть оплаты была бонусами (ECO.D.012), часть деньгами | Возврат денежной части через канал; бонусы — по правилам балансов, не деньгами | Раздельная обработка двух потоков; единое уведомление с разбивкой |
| **Refund после chargeback** | Плательщик инициировал chargeback в банке + параллельно запрос refund | Не возвращать дважды; приоритет — chargeback (банк уже забрал деньги) | Скоординировать с банком; если оба пройдут — взыскать с плательщика |
| **Refund после revenue sharing settle** | Платёж уже распределён (автор/instructor получили долю) | Вычитать из следующей выплаты stakeholder'а (см. DP.SC.112 chargeback-логика) | Уведомить stakeholder'ов; зафиксировать в ledger |
| **Канал не поддерживает refund** | TG Stars cashback — ограничения Telegram (только новые Stars обратно) | Возврат в форме «новых Stars», не в фиате | Уведомить плательщика заранее; альтернатива — баллы платформы |

## Связь с другими обещаниями

- **Использует [ECO.D.010](ECO.D.010-payment-models.md)** — правила refund различаются для подписки / покупки / доната.
- **Использует [ECO.D.011](ECO.D.011-retention-vs-cancel.md)** — refund по подписке порождает cancel-событие.
- **Использует [ECO.D.012](ECO.D.012-money-vs-balances.md)** — refund возмездных средств vs возврат бонусов — разные потоки.
- **Связано с [ECO.SC.001](ECO.SC.001-payment-receipt.md)** — refund использует тот же канал, что и платёж, и тот же SLA по банковским дням (T+N).
- **Связано с [DP.SC.112](../../../../../PACK-digital-platform/pack/digital-platform/08-service-clauses/DP.SC.112-subscription-billing.md)** — реализация cancel + refund Stars в боте; chargeback из revenue sharing.

## Не покрывает

- Точные значения X₁ / X₂ (это бизнес-политика, фиксируется в оферте, не в Pack).
- LTV-расчёт с учётом refund-rate (это v2).
- Налоговую обработку refund (возврат НДС, корректировки декларации — отдельный сервис).
