# [ECO.SC.001] Деньги поступили на счёт за N банковских дней

## Обещание

**Кому:** Получателю платежа (пилот как провайдер услуги / автор контента), Бухгалтеру, Финансовому контролёру.

**Зачем:** Прозрачность движения денег от платёжного шлюза до расчётного счёта. Превышение нормативного срока — управляемый инцидент, не «потерянный» платёж.

**Что получит:**
- Подтверждение факта оплаты от шлюза (webhook, статус `paid`) — немедленно (минуты).
- Зачисление средств на расчётный счёт — в течение **N банковских дней** (N зависит от канала, фиксируется в договоре с провайдером; типично N=1..7).
- Отчёт о расхождении: если фактическое зачисление > N — автоматический инцидент с темой «payment-settlement-overdue».

**Инвариант:** Деньги от плательщика попадают на расчётный счёт получателя не позднее N банковских дней с момента подтверждённой оплаты. Превышение → инцидент по DP.SC.121-каскаду уведомлений.

**Критерий приёмки:** За контрольный период (месяц) ≥99% платежей зачислены в SLA N. Каждое превышение зафиксировано инцидентом и закрыто с указанием причины.

## Основной flow

```
Плательщик (пилот) → Платёжный шлюз → Списание со счёта плательщика
       │                                       │
       │                                       ▼
       │                              Webhook → платформа: статус "paid"
       │                                       │
       │                                       ▼
       │                              Запись в finance.payments (T+0)
       │                                       │
       ▼                                       ▼
Уведомление "оплата принята" ← ────  Зачисление на расчётный счёт
                                       получателя (T+N банковских дней)
                                                │
                                                ▼
                                       Сверка: ожидается vs фактически
                                                │
                                                ├── совпало → OK, закрыть платёж
                                                └── расхождение → инцидент
```

## SLA N по каналам

> Конкретные значения N задаются в договорах с провайдерами. Здесь приведены типовые ориентиры (не нормативные).

| Канал (см. DP.SC.112) | Типовой N (банковских дней) | Юрисдикция |
|----------------------|-----------------------------|-----------|
| YooKassa | 1-3 | РФ (ИП) |
| Paybox | 1-3 | РФ (ИП) |
| Stripe | 2-7 | Мир (Aisystant Corp USA) |
| Монета / PayAnyWay | 3-7 | РФ+СНГ+Мир |
| TG Stars | 21-30 (Stars → фиат через Telegram cashout) | Глобальный |
| Manual (B2B) | По договору (5-30) | По юрлицу |

## Failure modes

| Класс | Симптом | Что происходит | Recovery |
|-------|---------|---------------|----------|
| **Банк отозвал транзакцию** | webhook `paid` → позднее `refunded` без действия получателя | Запись cancel-события (ECO.D.011), уведомление получателю и плательщику | Анализ причины (chargeback, fraud) → reply через канал спора |
| **Валютный контроль** | Stripe / международный канал → платёж заблокирован на N+M дней | Зачисление откладывается, статус `pending-verification` | Документы в банк / провайдер; SLA выставляется заново после разблокировки |
| **Неверный реквизит** | Зачисление не прошло, средства повисли у провайдера | Уведомление получателю «требуется уточнение реквизита» | Обновить реквизит в кабинете провайдера → повторное зачисление |
| **Retry-логика шлюза** | Шлюз пытается списать N раз → дубликаты webhooks | Идемпотентность по `idempotency_key` (см. lessons_idempotency_hash_fallback) | Сверка по уникальному ID транзакции; дубликаты игнорируются |
| **Превышение N** | Прошло N дней, зачисления нет | Автоматический инцидент `payment-settlement-overdue` | Эскалация в провайдера; параллельно — уведомление получателю с timestamp ожидаемого решения |

## Связь с другими обещаниями

- **Использует [ECO.D.010](ECO.D.010-payment-models.md)** — определяет, какая модель транзакции (подписка / покупка / донат) — это влияет на расчётный счёт и комиссию.
- **Использует [ECO.D.012](ECO.D.012-money-vs-balances.md)** — отделяет возмездные транзакции от учёта балансов; ECO.SC.001 покрывает только возмездные.
- **Связано с [DP.SC.112](../../../../../PACK-digital-platform/pack/digital-platform/08-service-clauses/DP.SC.112-subscription-billing.md)** — техническая реализация платёжного канала. DP.SC.112 покрывает «как получить webhook», ECO.SC.001 — «когда деньги физически на счёте».
- **Связано с [ECO.SC.002](ECO.SC.002-refund-policy.md)** — возврат за X (обратное движение по тому же каналу).
- **Связано с [DP.SC.121](../../../../../PACK-digital-platform/pack/digital-platform/08-service-clauses/DP.SC.121-ontology-grounded-answers.md)** — PII boundaries (плательщик идентифицируется ory_id, не email/телефоном).

## Не покрывает

- LTV / ARPU / cohort-аналитику (это v2, после WP-377 → полный PACK-business).
- Налоговый учёт (только факт зачисления, без проводок).
- Reconciliation с банковской выпиской (отдельный сервис).
