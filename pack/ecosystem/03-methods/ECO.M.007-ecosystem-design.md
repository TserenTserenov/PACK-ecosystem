# ECO.M.007: Ecosystem Design (API-first, Platform Thinking)

> **Назначение:** Метод проектирования экосистемы как связной системы компонентов с API-контрактами, сетевыми эффектами и точками расширения для третьих сторон.

---

## 1. Определение

Ecosystem Design — метод проектирования, при котором экосистема строится не как монолитный продукт, а как **система взаимодействующих акторов** вокруг общей платформы. Фундаментальный принцип: API-контракт первичен, реализация вторична.

**Различение:** Ecosystem Design ≠ System Architecture. Ecosystem Design определяет *кто с кем взаимодействует и зачем* (бизнес-уровень). System Architecture определяет *как технически реализованы взаимодействия* (инженерный уровень).

## 3. API-First как стратегия экосистемы

**Принцип:** API планируется, документируется и тестируется **до** написания кода приложения.

### Шаги

1. **Определить bounded contexts** экосистемы (компоненты, сервисы)
2. **Спроектировать API-контракты** между ними (OpenAPI, AsyncAPI для событий)
3. **Опубликовать контракты** до реализации — потребители начинают разработку параллельно
4. **Реализовать** — каждый компонент реализуется независимо за контрактом
5. **Мониторить** — API usage analytics показывают реальные паттерны взаимодействия

### API-First checklist

- [ ] Контракт задокументирован (OpenAPI/AsyncAPI)
- [ ] SDK/клиент генерируется из контракта
- [ ] Версионирование определено (URL path vs header)
- [ ] Rate limiting и SLA описаны
- [ ] Breaking changes → deprecation policy (минимум 1 цикл)

## 4. Platform Thinking: от продукта к экосистеме

**Прогрессия:** Product → Platform → Ecosystem

| Стадия | Ценность создаётся | Масштабирование | Пример (IWE) |
|--------|-------------------|-----------------|---------------|
| **Product** | Командой для пользователей | Линейно (больше фич = больше людей) | Курс «Системное мышление» |
| **Platform** | Командой + третьими сторонами через API | Субконтрактное (third parties добавляют ценность) | LMS + API для агентов |
| **Ecosystem** | Всеми участниками друг для друга | Сетевые эффекты (каждый участник повышает ценность для всех) | Методология + AI-агенты + Сообщество + Партнёры |

**Ключевой переход (Product → Platform):** Появление API или точки расширения, через которую третья сторона может создавать ценность без участия ядра.

**Ключевой переход (Platform → Ecosystem):** Появление взаимодействий *между* третьими сторонами без посредничества платформы.

---

## 5. Сетевые эффекты

| Тип | Механизм | Пример (IWE) |
|-----|----------|---------------|
| **Direct** | Больше пользователей → больше ценность для каждого | Клуб: больше участников → богаче дискуссии |
| **Indirect (cross-side)** | Больше одной стороны → больше ценность для другой | Больше студентов → больше менторов → лучше опыт |
| **Data** | Больше использования → лучше AI/персонализация | Больше прохождений → точнее Digital Twin |

**SOTA 2025-2026:** Компании с платформенной моделью получают **2-3× выше** оценку, чем standalone-продукты (network effects + retention).

## 6. Паттерны бесшовности (Seamlessness Patterns)

| Паттерн | Описание | Реализация |
|---------|----------|------------|
| **Single Sign-On** | Один аккаунт для всех компонентов | OAuth/OIDC через общий identity provider |
| **Unified Data Model** | Прогресс и контекст доступны из любого компонента | Shared data layer / API gateway |
| **Cross-component Navigation** | Переход между компонентами без потери контекста | Deep linking, state passing |
| **Event-driven Sync** | Изменение в одном компоненте → реакция в других | Webhooks, event bus (pub/sub) |

## 7. Применение к IWE

| Аспект | Текущее | Целевое |
|--------|---------|---------|
| Идентичность | Ory (общий identity) | ✅ Единая аутентификация |
| Данные | Neon (платформенная БД) + отдельные хранилища | API layer над данными, Digital Twin API |
| Интеграции | Бот как точка входа | Bot + Web + Agent SDK как точки входа |
| Third-party | Нет | Agent Marketplace, Partner integrations |

## 8. Failure modes

| ID | Ошибка | Последствие |
|----|--------|-------------|
| FM.ED.001 | Строить экосистему без продукта | Нет ядра ценности — нечего расширять |
| FM.ED.002 | API как afterthought | Нестабильные контракты, breaking changes, недоверие партнёров |
| FM.ED.003 | Over-design: проектировать ecosystem на стадии product | Complexity без пользователей, premature abstraction |
| FM.ED.004 | Игнорирование governance | Качество экосистемы падает, spam, low-quality integrations |
| FM.ED.005 | Закрытая платформа без extension points | Нет сетевых эффектов, линейное масштабирование |
