# ECO.M.025 — Outcome Tags for Support Log (Теги исхода в журнале поддержки)

## Определение

Метод структурирования журнала обращений участников экосистемы через обязательные теги исхода. Каждое обращение получает минимальный outcome-тег (`success` / `problem`) при закрытии.

## IPO

- **Вход:** закрытое обращение участника (вопрос, блокер, жалоба)
- **Процесс:** присвоить outcome-тег; при `problem` — категорию (опционально)
- **Выход:** структурированная запись, пригодная для агрегации

## Ключевое различение

**Без тегов:** журнал = архив, неискаемый по результату.  
**С минимальным тегом:** журнал = аналитический инструмент.

Аналогия со структурированным логированием в коде: free-text неискаем, tagged-record запрашиваем.

## Что даёт минимальный тег

После накопления N записей можно отвечать на вопросы:
- Какой % обращений заканчивается успехом?
- Какие problem-паттерны повторяются?
- В какой точке воронки (онбординг / активное участие / выход) больше всего проблем?

## Уровни детализации

| Уровень | Теги | Что позволяет |
|---------|------|---------------|
| Минимальный | `success` / `problem` | % успешных обращений |
| Расширенный | + категория проблемы | паттерны типов блокеров |
| Полный | + этап воронки, время разрешения | вороночный анализ |

## Применимо к

- Любому журналу обращений участников (онбординг, техподдержка, менторинг)
- Каналам обратной связи сообщества
- Внутренним helpdesk-системам без готовой аналитики
