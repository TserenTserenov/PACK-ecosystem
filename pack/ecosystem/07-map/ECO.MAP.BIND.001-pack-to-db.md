# [ECO.MAP.BIND.001] Мост Pack↔БД: системы продвижения → Neon

> Anti-corruption layer (WP-251 Ф2). Связывает type-level описания `ECO.SYS.*` с token-level таблицами Neon (`DP.ARCH.004-neon-data-architecture.md` v2.6). Легаси-имена БД в Pack не затаскиваются — Pack остаётся источником доменного словаря, эта карта только даёт mapping.

## Различение (напоминание, FPF A.7)

Pack-тип (`ECO.SYS.001-lead.md`) ≠ DB-таблица (`lead.LEAD`) ≠ DB-строка (конкретная запись с `lead_id`). Связь — только через явный маппинг ниже, не совпадение имён.

## Маппинг по системам

### ECO.SYS.001 Лид

| Состояние Pack | Таблица(ы) Neon | Комментарий |
|---|---|---|
| 1 холодный | `#11 lead.LEAD` (без `CLAIM`) | Строка создаётся landing/acquisition-funnel до регистрации в Ory |
| 2 прошёл первый контакт/диагностику | `#11 lead.LEAD` + `#11 lead.FUNNEL_RECORD` (`event_type=activation`) | Прогресс по воронке фиксируется как запись, не смена статуса LEAD |
| 3 покупатель (граничное) | `#11 lead.CLAIM` + `#3 payment.PAYMENT` (`status=succeeded`) | Композитное событие, не отдельная строка с собственным жизненным циклом (см. §Гап-матрица, п.1) |

Метод `ECO.M.022-lead-to-buyer` = запись `CLAIM` (lead_id→account_id) с последующим первым успешным `PAYMENT`. После `CLAIM` PII лида мигрирует в `#1 persona.ACCOUNT`, остаток (`UTM_VISIT`, `FUNNEL_RECORD`) остаётся в `#11` для аналитики воронки — согласовано с инвариантом `#11` (DP.ARCH.004 §3.11).

### ECO.SYS.002 Сообщество

| Состояние Pack | Источник Neon | Комментарий |
|---|---|---|
| 1 пассивная база | Агрегат `#1 persona.ACCOUNT`, у которых есть `CLAIM`, но нет активного `#4 subscription.CONTRACT` | Derived, не отдельная таблица |
| 2 активное ядро формируется | Растущая доля `ACCOUNT` с `CONTRACT.status=active` | Derived-метрика |
| 3 активное ядро | Стабильная когорта `ACCOUNT` с `CONTRACT.status=active` + участием в `#10 community` (MENTORSHIP/GROUP) | Derived-метрика |

Прямой таблицы «Сообщество» в Neon нет — коллективное состояние ECO.SYS.002 полностью производное (см. §Гап-матрица, п.2), что согласуется с собственным инвариантом Pack-файла («Сообщество не имеет индивидуального состояния... агрегатная система»).

### ECO.SYS.003 Член сообщества

| Состояние Pack | Таблица(ы) Neon | Комментарий |
|---|---|---|
| 1 базовый член | `#1 persona.ACCOUNT` существует (через `CLAIM`), `#4 subscription.CONTRACT` отсутствует или неактивен | Прямого поля-статуса нет — состояние читается по отсутствию активного контракта |
| 2 активный подписчик | `#4 subscription.CONTRACT` (`status=active`) | Точное совпадение — подтверждает уточнение WP-385 «подписная модель Мастерской», уже зафиксированное в `ECO.SYS.003.md §States` |
| 3 наставник | `#10 community.MENTORSHIP` (`mentor_account_id = account_id`, роль подтверждена Keto-грантом) | Точное совпадение |
| 4 мастер | Нет прямой таблицы — ближайшие косвенные факты: `#10 community.INITIATIVE` (автор) / `AMBASSADORSHIP` | Гап (см. §Гап-матрица, п.3) |

## Гап-матрица

| # | Находка | Что в Pack | Что в БД | Решение |
|---|---|---|---|---|
| 1 | «Покупатель» — не персистентная сущность | Граничное состояние `ECO.SYS.001` (переход 2→3) | Нет отдельной таблицы; composite-событие `CLAIM`+`PAYMENT(succeeded)`, немедленно сменяется `#4 subscription.CONTRACT` | **`ECO.SYS.004-buyer` НЕ создаётся.** БД не содержит «осиротевшей» таблицы без места в SYS.001/002/003 — единственный кандидат («покупатель») подтверждён как переходное событие, не персистентный объект. Решение WP-251 Ф1 (отложить до эмпирики Ф2) закрывается отрицательно. |
| 2 | «Сообщество» как коллективная система — derived, не stored | `ECO.SYS.002` описывает состояния 1-3 как атрибуты системы | Нет таблицы-агрегата; состояние вычисляется по `ACCOUNT`+`CONTRACT`+`#10 community` | Не гап, а подтверждение: Pack-файл сам называет эти состояния «агрегатной метрикой», не индивидуальным жизненным циклом — mapping это лишь фиксирует явно. Кандидат Ф5: materialized view/метрика для операционного использования Алёной (не архитектурное решение, вопрос удобства). |
| 3 | «Мастер» (состояние 4, известность за пределами Сообщества) — нет прямого DB-эквивалента | `ECO.SYS.003 §States` состояние 4 | `#10 community.INITIATIVE`/`AMBASSADORSHIP` фиксируют факты, не флаг «известности» | Кандидат Ф5 (уже отмечено в Pack как некандидат Ф1 — «нет эмпирического материала кроме WP-385 уровня 4»). Мост не решает это на Ф2 — недостаточно данных в БД, не архитектурный вопрос. |
| 4 | «Вступительный разбор костюма» (WP-385, гейтинг качества) | Упомянут в WP-251.md как режим `ECO.PIPE.001`, не состояние | Нет отдельной таблицы в `#3`/`#4`/`#10` — вероятно частный случай `#4 subscription.CONTRACT` с `activating_payment_id`, ведущим не через обычную воронку | Не блокирует Ф2 — кандидат Ф5 вместе с методами `1→2→3→4` ECO.SYS.003. |

## Итог по «отдельному типу Покупатель»

Вопрос, отложенный на Ф1 (`ECO.SYS.001.md §Boundary states`, `ECO.M.022.md §Открытые вопросы`), закрывается: **новая Pack-система не нужна**. Гап-матрица не обнаружила таблицу Neon, для которой существующие `ECO.SYS.001/002/003` не дают естественного места. «Покупатель» остаётся именованным граничным состоянием ECO.SYS.001, реализуемым парой событий `CLAIM`+`PAYMENT(succeeded)` из `#11 lead` и `#3 payment` — ровно так, как уже сформулировано методом `ECO.M.022-lead-to-buyer`. Симметрия «3 системы + 2 конвейера», заложенная на Ф0-Ф1, подтверждена эмпирически.
